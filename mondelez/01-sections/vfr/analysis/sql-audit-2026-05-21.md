## Greenlight Verdict (pre-fix) вАФ рЯФі BLOCK Phase 7 Ship

**Reason:** 14 of 16 SQL sections return EMPTY data under both Scenario A (first-paint, all-empty filters) and Scenario B (narrow filter with 1 active multi-select). Root cause is a critical interaction bug between `WidgetFilterResolver.EscapeAsSqlLiteral` (returns literal `"NULL"` for empty values) and the canonical `if(arraySort([{{x}}]) = (SELECT arraySort(groupArray(DISTINCT ...)) FROM mv_filter_*), 1=1, t.col IN ({{x}}))` membership-probe pattern. When **any** multi-select filter is empty, the probe evaluates to FALSE вЖТ fallback to `t.col IN (NULL)` вЖТ entire WHERE excludes all rows. Only sections WITHOUT the filter block (vehicleTypeTender/OpsOptions, plus KPI scalars that return a phantom NULL/0 row) survive.

Saturation probe (Scenario X вАФ all 4 multi-selects sent as the full distinct sets from the filter MVs) confirms the SQL **algebra** is correct (11 groups, 2467 rows, total CBM вЙИ 38826.66 over `mv_vfr_gui_thau`). The bug is exclusively in **empty-filter encoding** вАФ the resolver does not treat array-context placeholders specially.

## 6. Pattern compliance report

### 6.1 CRITICAL вАФ `arraySort([{{x}}])` probe broken for empty filters (14 of 16 sections)

`WidgetFilterResolver.EscapeAsSqlLiteral("")` returns the literal string `"NULL"` (verified in `WidgetFilterResolverTests.cs` L60-68 and `WidgetFilterResolver.cs` L136). Substituted SQL becomes:

```sql
if(arraySort([NULL]) = (SELECT arraySort(groupArray(DISTINCT vendor_code)) FROM analytics_workspace.mv_filter_vendor),
   1 = 1,
   t.nha_van_tai IN (NULL))
```

Probe evaluation:
- `arraySort([NULL])` = `[NULL]` (Array(Nullable(Nothing)))
- Equality with `['ANH SON','GHN','HDA','HOA PHAT','HVP','NGUYEN PHAT','NINJAVAN','NJV- NhбЇ•t T√≠n','THANH AN','TLL','VAN HO']` вЖТ **FALSE** (verified empirically)
- IF takes FALSE branch вЖТ `t.nha_van_tai IN (NULL)` вЖТ NULL вЖТ row excluded.

**Impact:** Every section that includes the 4-filter block (all 14 of 16 sections except `vehicleTypeTender/OpsOptions`) returns zero data on **first-paint** (`filterOverrides = { whseid:'', area:'', transporter:'', vehicle_type_tender:'', vehicle_type_ops:'' }` per `widget-vfr.tsx` L733-756). Even one un-set multi-select kills the entire query.

**This is the Phase 7 ship-blocker.**

**Remediation options** (ranked):

| Option | Where | Effort | Notes |
|---|---|---|---|
| **A вАФ Backend resolver patch** (recommended) | `WidgetFilterResolver.EscapeAsSqlLiteral` | S | Detect bracket-context `[{{x}}]` and, on empty, emit the saturated set from a registered filter MV вАФ or simpler, emit a special sentinel that the probe pattern can match. Simplest: when value is empty and placeholder appears inside `[вА¶]`, emit the literal contents of the matching filter MV via a parametrized lookup. |
| **B вАФ Pattern rewrite at registry** | `sql-registry.md` √Ч 14 blocks + spec ¬І22 √Ч 2 | M | Replace `if(arraySort([{{x}}]) = (...), 1=1, t.col IN ({{x}}))` with `if({{x}} IS NULL, 1=1, t.col IN ({{x}}))`. Drops the filter-MV round-trip; relies on the resolver substituting bare `NULL` for empty values (already does вАФ `EscapeAsSqlLiteral("")` вЖТ `"NULL"`). Caveat: needs verification that ClickHouse parses `NULL IS NULL` correctly (it does вАФ returns 1). |
| **C вАФ FE emits saturated CSV when "ALL"** | `widget-vfr.tsx` filterOverrides memo (L733) | M | Pass the resolved CSV (`pickupWarehouse.join(',')`) instead of `''` when "ALL" is selected. Couples FE to master-data, fragile when filter MV is refreshed. Not recommended. |
| **D вАФ Drop multi-select; revert to single-select + `coalesce({{x}}, 'ALL') = 'ALL'`** | All registry SQLs | L | Loses Phase 3 multi-select capability. Not acceptable. |

**Recommended decision:** Option B (pattern rewrite) вАФ touches 16 SQL blocks but no backend code change. Smaller blast radius, no resolver semantics shift, no FE coupling.

### 6.2 HIGH вАФ `{{vehicle_type_tender}}` placeholder used in 7 operation SQLs (should be `{{vehicle_type_ops}}`)

Registry ¬Іvfr operation hardcodes `arraySort([{{vehicle_type_tender}}]) вА¶ t.ma_loai_xe_van_hanh IN ({{vehicle_type_tender}})` across all 7 operation SQL blocks (L5517, 5616, 5716, 5814, 5913, 6167, 6463, 6702, 6940, 7131). Spec ¬І22.2 byVendorOperation correctly uses `{{vehicle_type_ops}}`.

**Impact under live traffic:** When the user is in operation mode and applies a `vehicle_type_ops` filter, the SQL parameter substitution misses (because the placeholder name is `vehicle_type_tender`, not `vehicle_type_ops`). The operation widget filters by the **tender** vehicle type the user picked while in tender mode (sticky state in `applied.vehicleTypeTender`), or by nothing if tender mode was never visited. Functional bug, but Scenario A/B used empty vehicle filters so the runtime symptom in this audit is masked.

**Fix:** Sed rewrite in `sql-registry.md` for the 7 operation blocks: `{{vehicle_type_tender}}` вЖТ `{{vehicle_type_ops}}`. Trivial, no algebra change.

### 6.3 OK вАФ date filter outside CTE, params CTE / inline params, single-quote escape

All 16 SQLs put the date clamp `BETWEEN toDate(coalesce({{from_date}}, '1900-01-01')) AND вА¶` in the inner CTE WHERE, OUTSIDE any params CTE вАФ matches canonical pattern (memory `feedback_sql_review_widget_runtime`). No `coalesce({{multi_select}}, 'ALL')` anti-pattern detected.

Single-quote escape (`O''Brien`) handled by resolver (`WidgetFilterResolverTests.cs` L70-94) вАФ no injection surface from filter values.

### 6.4 OBSERVATION вАФ Vietnamese column names work but invite churn

Detail SQLs lean on FE alias fallback to Vietnamese keys (`"ID chuyбЇњn gбї≠i thбЇІu"`, `"TбЇ•n nhбЇ≠n"`, вА¶). FE handles via `vfrSqlS` case-insensitive lookup. Functional, but: (a) any future MV column rename in upstream pipeline breaks silently вАФ alias matrix update needed in 2 places (registry + FE), and (b) JSON serialization of Vietnamese keys with diacritics adds bandwidth/parse cost on a 5,000-row detail payload. Low priority; flag for future cleanup.

## Fix Verification (2026-05-21 post-fix)

**Fix:** Option B вАФ `/debugger` rewrote 80 arraySort membership probes in `sql-registry.md` (lines 3721-7152, ¬Іvfr tender + ¬Іvfr operation) and 5 in `vfr-spec.md` (¬І22.1 + ¬І22.2) from
```
AND if(arraySort([{{x}}]) = (SELECT arraySort(group*Array(... )) FROM analytics_workspace.mv_filter_*), 1=1, t.col IN ({{x}}))
```
to
```
AND (coalesce({{x}}, 'ALL') = 'ALL' OR t.col IN ({{x}}))
```
Also renamed `{{vehicle_type_tender}}` вЖТ `{{vehicle_type_ops}}` in 10 occurrences inside ¬Іvfr operation (where filter targets `t.ma_loai_xe_van_hanh`).

**Why `coalesce({{x}}, 'ALL') = 'ALL'`, not `{{x}} IS NULL`:** CSV substitution emits `'HVP', 'HDA'` (no parens). `if('HVP', 'HDA' IS NULL, ...)` parses as 4 arguments to `if()` вЖТ `Code: 42 NUMBER_OF_ARGUMENTS_DOESNT_MATCH` (verified empirically). `coalesce('HVP', 'HDA', 'ALL')` parses cleanly as variadic вАФ returns the first non-NULL ('HVP') вЖТ `'HVP' = 'ALL'` вЖТ FALSE вЖТ falls through to `IN` clause. Works for all 3 resolver substitution forms.
