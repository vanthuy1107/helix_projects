## BUG-VFR-09 — Aggregate SQL `AVG(vfr_max)` simple, không khớp TOBE V3 Bước 5

**Severity:** Major (data accuracy)
**Status:** **Open** — discovered 2026-05-20 sau khi đọc `[Version 3.0] TOBE - GIẢI PHÁP CONTROL TOWER.pdf` (slide 50, line 989-993).
**Affected SQL:** `kpi.avg_vfr`, `kpiOperation.avg_vfr`, có thể cả `byArea`, `byVehicle`, `byLoadingType`, `byTimeArea` (tất cả aggregate sections trong sql-registry §7-8)

### Evidence

V3 spec (slide 50, Bước 4-5) yêu cầu aggregate theo công thức Loose+FP weighted:

```
Bước 4 (per loại bốc xếp):
  VFR loại bốc xếp 1 = VFR Tấn(c) × (Σ Khối chở Tấn / Σ Khối đk Tấn)
                       + VFR Khối(d) × (Σ Khối chở Khối / Σ Khối đk Khối)

Bước 5 (tổng):
  VFR Tổng = Σ [VFR loại bốc xếp n × (Σ Khối chở LT_n / Σ Khối đk LT_n)]
```

Widget tooltip `VFR_FORMULA` tại [widget-vfr.tsx:201](../../../../../frontend/src/features/dashboard/components/widgets/order-monitor/widget-vfr.tsx#L201) cũng echo công thức này.

SQL thực tế trong [sql-registry.md §3590](../../../02-data/data-sources/sql-registry.md#L3590):
```sql
SELECT AVG(`vfr_max`) AS avg_vfr FROM analytics_workspace.mv_vfr_gui_thau ...
```

→ Đây là `AVG(per-trip MAX)` simple, KHÔNG phải weighted aggregate V3 Bước 5.

V3 line 995 chỉ rõ: **"THAM KHẢO: File Data mẫu tính % Truck Utilization theo loại xe gửi thầu"** — file mẫu là entry point để data team verify.

### Impact

- KPI card "Avg VFR" hiển thị giá trị **không khớp spec Mondelez TOBE 3.0**.
- Mọi chart aggregate ("VFR theo Khu vực", "theo Loại xe", "theo Loại bốc xếp", "Time × Area") có thể đang dùng simple aggregate thay vì weighted.
- Audit Mondelez hỏi "Avg VFR đo theo công thức nào?" — câu trả lời hiện tại là `AVG(vfr_max)`, không phải Bước 5 → mất uy tín dashboard.

Per-trip `vfr_max` và bucket counting (`cnt_vfr_50`, `cnt_vfr_50_70`, etc.) KHÔNG bị ảnh hưởng — chúng tính trên per-trip values, không phải aggregate weighted.

### Fix candidates

- **A (recommended):** Re-write SQL aggregate cho `kpi.avg_vfr` + `byArea` + `byVehicle` + `byLoadingType` + `byTimeArea` theo V3 Bước 4-5. Cần CTE phức tạp với multiple SUM/SUM ratios. Data team owner.
- **B (alternative):** Nếu business chấp nhận `AVG(vfr_max)` là proxy đủ tốt cho Mondelez ops, update widget tooltip nói thật là dashboard dùng simple AVG (không theo V3 Bước 5). Documentation-only fix.

User decision (2026-05-20): **A** — implement đúng V3 spec.

### Rollout

Tương tự BUG-VFR-08:
1. Data team viết SQL mới khớp V3 Bước 4-5 (reference V3 line 995 file mẫu).
2. Update sql-registry.md.
3. Tenant admin re-paste SQL vào WidgetVfrSettingsDialog cho Mondelez.

### Owner

Data team — registry rewrite + tenant config rollout.

### Open sub-questions

- Per-dimension aggregate (per area, per vehicle): V3 không explicit. Có phải apply Bước 4-5 scoped per dimension không, hay đơn giản hơn? Data team confirm.
- Tooltip widget `VFR_FORMULA`/`VFR_NOTE` đang chỉ echo Bước 5 cho tổng. Per-dimension chart cần tooltip riêng?

## BUG-VFR-08 — CH `cnt_vfr_95` boundary `> 95` thay vì `>= 95`

**Severity:** ~~Major~~ (data accuracy)
**Status:** **Fixed in registry on 2026-05-19** (sql-registry.md tender + operation); tenant config rollout pending
**Discovered:** 2026-05-19 (BA-review reconcile)
**Affected SQL:** `mv_vfr_gui_thau` (tender) + `mv_vfr_van_hanh` (operation)

### Evidence

ClickHouse SQL trong `sql-registry.md`:
- Tender [sql-registry.md §3865](../../../02-data/data-sources/sql-registry.md#L3865): `AND vfr_max > 95`
- Operation [sql-registry.md §5298](../../../02-data/data-sources/sql-registry.md#L5298): `AND vfr_max > 95`

Redshift SQL (canonical reference) dùng `>= 95` đúng spec PRD gốc:
- Tender [sql-registry.md §3831](../../../02-data/data-sources/sql-registry.md#L3831): `AND "%VFR_theo_ID_chuyen_gui_thau(Max)" >= 95`
- Operation [sql-registry.md §5266](../../../02-data/data-sources/sql-registry.md#L5266): `AND "vfr_max" >= 95`

`cnt_vfr_70_95` dùng `< 95` (strict) → row `vfr_max = 95.00` exactly:
- Không match `vfr_max < 95` (bucket 70-95)
- Không match `vfr_max > 95` (bucket ≥95)
- → bị drop khỏi cả 2 bucket → không được đếm

### Impact

- KPI card "Excellent ≥95%" hiển thị count thấp hơn thực tế.
- KPI card "High 70-95%" cũng không bù lại được vì bị strict `< 95`.
- Tổng `Low + Medium + High + Excellent ≠ tổng số chuyến`.
- Avg VFR vẫn đúng (chỉ tính `AVG(vfr_max)`).

### Fix applied (2026-05-19)

Đổi `> 95` → `>= 95` trong 2 SQL CH (đồng thời xoá 2 stale comment lines `-- AND vfr_tmax < 95` / `-- AND vfr_max < 50` để giảm noise):
- `sql-registry.md` tender §Excellent CH section ([sql-registry.md:3865](../../../02-data/data-sources/sql-registry.md#L3865))
- `sql-registry.md` operation §Excellent CH section ([sql-registry.md:5296](../../../02-data/data-sources/sql-registry.md#L5296))

### Rollout — pending

Registry fix là single-source-of-truth, nhưng widget code đọc SQL từ `WidgetVfrSettingsDialog` config được persist trong DB per-tenant. Cần:
1. Tenant admin của Mondelez mở dashboard VFR → click "Setting Chart" → paste SQL mới từ registry cho section `kpi` (tender) + `kpiOperation` (operation) → Save.
2. (Hoặc) data team chạy migration script update widget config cho Mondelez tenant trực tiếp trong DB.

Cho tới khi rollout xong, **production vẫn còn bug** — chỉ registry đã correct.

### Verification

- ✅ Grep toàn repo: chỉ 2 occurrences trong `sql-registry.md`, không có duplicate ở backend/frontend code.
- ✅ Diff confirmed `> 95` → `>= 95` ở cả 2 SQL CH; bucket `cnt_vfr_70_95` giữ nguyên `< 95` strict (đúng).
- ⏳ Functional verify (sau rollout): chạy query test với row `vfr_max = 95.0` exactly → kiểm tra count xuất hiện ở `cnt_vfr_95`. Lý tưởng nhất là tạo unit test SQL fixture (CH thiếu native test framework — cân nhắc record-mode test ở /tester).

3. Render Alert (variant=destructive) ở top của widget body khi `sqlApiData.errors.length > 0`, hiển thị count + retry button.
4. Retry button gọi `refetchSqlApi()`, show spinner khi `isSqlFetching`.
5. Added i18n keys: `orderMonitor.vfr.dataFetchError` (vi+en, with {{count}} interpolation) + `orderMonitor.vfr.retry` (vi+en).
